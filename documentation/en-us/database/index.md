# Database Driver Design & New Driver Guide

The core goal of RBQ (Rust Business Query) is to provide a unified, secure, and high-performance execution interface for different databases. This document outlines the unified design principles for RBQ drivers and provides guidance on how to add a new database driver.

## 1. Unified Design Principles

To ensure consistent behavior across all drivers, RBQ follows these design principles:

### 1.1 Protocol Respect
RBQ drivers should be implemented based on the native protocols of the databases (e.g., MySQL binary protocol, PostgreSQL V3 protocol). We prefer native protocol implementations over generic ODBC/JDBC bridges to achieve maximum performance and native feature support.

### 1.2 Mandatory Parameter Binding
All drivers must execute SQL via parameterized queries (Prepared Statements).
- **Prohibited**: String concatenation at the driver level.
- **Required**: Transmit SQL templates and parameters (`Value` arrays) separately to the database engine.
This fundamentally eliminates SQL injection risks.

### 1.3 Unified Error Model
Drivers are responsible for mapping errors generated by the underlying protocol (ErrorCode, SQLSTATE) to `rbq_types::RBQError`.
- Common mappings include: `ConnectionError`, `DatabaseError`, `AuthenticationError`, etc.

### 1.4 Transparent State
Drivers must accurately report the `ConnectionState` (e.g., `Idle`, `InTransaction`, `Busy`) of a connection so the connection pool can correctly manage its lifecycle.

## 2. New Driver Guide

To add support for a new database (e.g., `we-trust-oracle`), follow these steps:

### Step 1: Create Package
Create a new crate in the `yyds/packages/` directory named `we-trust-<name>`.

### Step 2: Implement Connection Trait
Implement the `rbq_core::Connection` interface, which is the heart of the driver.
```rust
pub trait Connection: Send + Sync {
    type Tx: Transaction;
    async fn execute(&self, query: &str, params: &[Value]) -> RBQResult<ExecuteResult>;
    async fn query(&self, query: &str, params: &[Value]) -> RBQResult<ResultSet>;
    // ... other methods ...
}
```
**Key Points**:
- Ensure `execute` and `query` use asynchronous I/O.
- Correctly handle parameter binding logic.

### Step 3: Implement DriverAdapter Trait
Implement `rbq_core::DriverAdapter`, which is responsible for creating connections.
```rust
pub trait DriverAdapter: Send + Sync + 'static {
    type Conn: Connection;
    async fn connect(&self, config: &DbConfig) -> RBQResult<Self::Conn>;
    // ... metadata methods ...
}
```

### Step 4: Define Type Mappings
Implement bidirectional conversions between native database types and `rbq_types::Value` within the driver.

### Step 5: Error Mapping
Write a helper function or implement the `From` trait to convert underlying error types into `RBQError`.

### Step 6: Testing & Verification
- **Unit Tests**: Test protocol parsing and type conversions.
- **Integration Tests**: Write connection and query tests against real database instances in the `tests/` directory.

## 3. Best Practices

- **Zero-Copy Parsing**: Use libraries like `bytes` or `nom` for high-performance protocol parsing whenever possible.
- **Async First**: Underlying I/O must use `tokio` or a similar asynchronous runtime.
- **Extensibility**: Provide extended support for database-specific features (e.g., Redis Pub/Sub) through `DriverExtension`.
